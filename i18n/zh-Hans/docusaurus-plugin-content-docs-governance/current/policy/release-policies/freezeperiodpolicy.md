# openRuyi 冻结期政策

openRuyi 作为滚动发行版，每个月底都会发布一个新的镜像；这样新用户只需要下载最新的镜像就可以体验最新的更改，免去了只有一个镜像用户还要进行大规模升级的不便。

每个月，openRuyi 的冻结过程分为两个阶段，这些会在下面进行进一步的阐述:

|          | 过渡      | 冻结         |
| -------- | ------- | ---------- |
| 时间       | 每个月的前两周 | 每个月的后两周    |
| 可接受的更改   | 海纳百川    | 小规模、针对性的修复 |
| 对工具链造成更改 | ✅       | ❌          |
| 添加/删除包   | ✅       | ❌          |

## 过渡期

过渡期一般为每个月的前两周，在这个阶段，我们会正常的接收任何更改，包括工具链的更新、bug 修复、软件包的添加，删除或更新等。

如果您计划进行大型或者破坏性的变更，请尽早进行自我评估来确定在冻结期之前完成这些变更是否现实。例如，您需要与其他人进行协作，但对方可能不一定有时间或者精力来参与这项工作。

## 冻结期

冻结期一般为每个月的后两周，在这个阶段，我们会专注于月底镜像的打磨，从而通常不会接收新的更改，例如:

* 工具链部分的软件包: 这类包出现问题时，解决通常需要花费大量的时间，因为除了 bug 修复本身，还需要搞清楚哪些包受了影响并需要重新构建。

* 大规模/破坏性更改: 这个阶段，我们会回避任何大型或破坏性的、或存在较高回归风险的更改。例如，某个库的升级除了会带来 ABI 更改之外，还发现了潜在的 bug，此时可能会阻碍其它无关修复的进行。

当然，对于严重 bug 的针对性修复我们是允许的。这里的 "针对性修复" 是指，针对某个 bug 而必须做出的最小限度修改。以下是在进行针对性修复时我们不推荐的行为:

* 升级为新的上游版本

* 在不同的软件包之间移动文件位置

* 修改软件包之间的依赖关系

* 进行可能影响到其他包的改动

* 修复的同时重构/清理代码

* 新增/移除或者重命名软件包
